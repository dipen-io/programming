8. database sharding ? how it is different from replication
    DB Replication -> copy the same data across multiple serves
        * one primary server (handle writes)
        * multple replica server (handle read)
        * purpose -> high availability  & read performace
        * if primary goes down -> replica takes over
    DB sharing -> split data across multple server 
        * each server hold different chunk of data 
        * example : Users A-M -> Server 1, User N-Z -> Server 2
        * purpose handle massive amounts of data that won't fit on one server 
    key difference
    Replication -> same data on multiple servers (for availability) 
    Sharding -> different data on each server (for scalibility)
9. What is the N+1 query problem in databases? How would you solve it?

    N+1 Query Problem:
        Imagine you have Posts and each post has an Author.
            // 1 query to get all posts
            const posts = await Post.find() // 1 query

            // Then for EACH post, you query the author separately
            posts.forEach(post => {
                const author = await User.findById(post.authorId) // N queries
            })

        if u have 100 posts -> 1+100  = 101 queries
            That's the N+1 problem -> too many unecessay DB queries

        how to fix : 
            option 1 : JOIN / Populate — fetch everything in a single query
            const posts =  await Post.find().populate('author'); 

            option 2 : Eager Loading — fetch all authors at once
            const authors = await User.find({ _id: { $in: posts.map(p => p.authorId) } })
            // 1 query instead of N

10 What is the CAP theorem? Explain each property and why it matters in distributed systems.
    C: Consistency
        * Every request gets the most recent data 
        * All servers return the same at the same time 
        * Example: update profile -> everyone seen the updated profile instanly

    A: Availability
        * System is always up and responding
        * Every request gets a resposne (even if it's not the latest data)
        * Example: Even if a server is struggling, it still responds

    P: Partition Tolerance 
        * System keeps working even if network connection between servers breaks 
        * In real distributed systems, network failure WILL happen - so P is basically mandatory

    Since P is always required, choose between
        Choice  Means                                      Example
        CP      Consistent but may be unavailable         Mongodb,  HBase
        AP      Always available but may have stale data  Cassandra,DynamoDB

    Simple analogy:
        During a network failure between two ATMs — you either:
            CP → block transactions until consistent (no wrong balance)
            AP → allow transactions but risk showing wrong balance
